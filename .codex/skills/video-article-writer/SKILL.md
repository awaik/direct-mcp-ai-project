# Video Article Writer — статья из видео

Полный цикл: видео → аудио → расшифровка → семантика для веб-статьи → статья 8000+ символов → анти-AI → редактура → SEO-пакет → горизонтальная обложка через LidFly.

Перед финальным контентом всегда сверяйся с `LEGAL.md`. Если тема регулируемая, спорная или медицинская, исправляй рискованные формулировки до сдачи.

## Главные правила

- Не выдумывай факты, которых нет в видео. Если нужен внешний факт, пометь его как требующий проверки или проверь через надёжный источник, когда это нужно для точности.
- Сохраняй авторский смысл видео, но не делай механическую расшифровку: статья должна читаться как самостоятельный веб-материал.
- Минимум статьи: 8000 символов с пробелами. Цель для сильной статьи: 9000-12000 символов, если тема позволяет.
- Считай и показывай оба числа: символы с пробелами и без пробелов.
- Обложку делай только через MCP LidFly `generate_ad_image`. Не используй `image_gen`, SVG, Canvas, локальное наложение текста или Magick для создания финальной картинки, если пользователь отдельно не попросил ручную доработку.
- Для обложки с надписями подавай весь промпт сразу в LidFly. Учитывай, что генератор может исказить кириллицу: делай короткие крупные надписи, но не переходи на ручное наложение без явного запроса.

## Входные варианты

1. Если дан видеофайл `.mov`, `.mp4`, `.mkv`, `.webm`, `.avi` — извлеки аудио и расшифруй.
2. Если дан аудиофайл `.mp3`, `.wav`, `.m4a`, `.aac`, `.ogg` — загрузи его на расшифровку напрямую.
3. Если дана готовая расшифровка `.txt` или `.md` — пропусти извлечение аудио и распознавание, используй текст как источник.
4. Если рядом уже есть `RESULTS/<basename>-transcription.txt`, предложи переиспользовать его вместо повторного распознавания.

## Шаг 1. Извлечь аудио

Для видео используй `ffmpeg`:

```bash
ffmpeg -i "<video_path>" -vn -acodec libmp3lame -ab 96k -ar 16000 -ac 1 -y "/tmp/<basename>.mp3"
```

Если `ffmpeg` не установлен, сообщи, что нужен `ffmpeg`, и предложи `brew install ffmpeg`.

## Шаг 2. Расшифровать через LidFly

Если нужные audio tools не загружены, сначала найди их через `tool_search` по запросу `lidfly audio transcription request_upload_audio get_transcription`.

1. Запроси upload URL:

```js
mcp__lidfly__request_upload_audio({
  filename: "<basename>.mp3",
  title: "<короткое название видео>",
  language: "ru",
  diarize: false
})
```

Для интервью или диалога поставь `diarize: true`.

2. Загрузи файл:

```bash
curl -s -T "/tmp/<basename>.mp3" "<upload_url>"
```

3. Получи текст:

```js
mcp__lidfly__get_transcription({ id: "<transcription_id>" })
```

Если статус `processing`, проверь ещё раз через одну итерацию. Не запускай бесконечные sleep-loop.

4. Сохрани расшифровку:

```text
RESULTS/<basename>-transcription.txt
```

## Шаг 3. Вытащить смысл видео

Перед семантикой сделай короткий контент-бриф:

- тема и обещание статьи;
- для кого материал;
- ключевая проблема читателя;
- главный вывод видео;
- 5-10 тезисов, которые точно должны попасть в статью;
- спорные или неполные места, которые нельзя превращать в уверенные факты;
- авторский тон: учебный, экспертный, разговорный, продающий, обзорный.

Сохрани бриф в:

```text
RESULTS/<basename>-article-brief.md
```

## Шаг 4. Собрать семантическое ядро для веб-статьи

Используй отдельный skill `semantic-core` в режиме `seo_site_core`.

Задача семантики здесь не рекламная кампания, а веб-статья:

- главный информационный ключ для H1 и первого абзаца;
- 5-10 поддерживающих ключей для H2/H3;
- 10-25 long-tail запросов для FAQ, примеров и подзаголовков;
- смежный спрос, который можно раскрыть одним-двумя абзацами;
- стоп-темы и нерелевантные интенты, которые нельзя тащить в статью;
- интент каждого кластера: обучающий, сравнительный, how-to, инструментальный, коммерческий.

Используй Wordstat через MCP `yandex-direct`, когда доступен. Если Wordstat недоступен, явно пометь семантику как оценочную.

Сохрани результат:

```text
RESULTS/<basename>-semantic-core.md
```

## Шаг 5. Построить структуру статьи

Перед написанием собери план, который покрывает и видео, и спрос:

- `H1` с главным ключом и понятной выгодой;
- лид 300-600 символов, главный ключ в первых 100 словах;
- 6-10 H2, каждый закрывает отдельный интент;
- H3 только там, где нужна детализация;
- блок "Что важно запомнить" или "Коротко";
- FAQ 4-7 вопросов из long-tail семантики;
- SEO title до 70 символов;
- meta description 140-160 символов;
- slug латиницей;
- 3-7 тегов или категорий.

Проверь план на каннибализацию: один H2 = один смысловой блок, без дублей.

## Шаг 6. Написать черновик

Пиши как редактор, а не как стенографист.

Обязательные требования:

- минимум 8000 символов с пробелами;
- раскрыть тему без воды и механических повторов ключей;
- использовать тезисы видео как основу экспертности;
- добавить примеры, сценарии применения, типичные ошибки, сравнения, мини-чеклисты;
- встроить ключи естественно, без переспама;
- не начинать текст клише вроде "в современном мире", "важно отметить", "следует подчеркнуть";
- не обещать гарантированный результат;
- если статья обучающая, дать читателю последовательность действий;
- если в видео есть демонстрация интерфейса, превратить её в понятные шаги;
- не использовать Markdown-таблицы без необходимости: таблица должна реально помогать сравнению.

Сохрани черновик:

```text
RESULTS/<basename>-article-draft.md
```

## Шаг 7. Убрать AI-маркеры

Применяй skill `ai-markers-remove`, но держи публикационное качество:

- убери гладкие AI-переходы, симметрию, однотипные абзацы;
- добавь живой синтаксис и авторские повороты;
- не ломай ключевые фразы, факты, заголовки, названия брендов и CTA;
- не оставляй грубые орфографические ошибки в финале;
- если skill внёс слишком заметные опечатки, исправь их на редакторском шаге.

Сохрани версию:

```text
RESULTS/<basename>-article-final.md
```

## Шаг 8. Редакторское ревью

Сделай ревью как редактор перед сдачей и внеси применимые правки.

Проверь:

- статья отвечает интенту главного ключа;
- первые 2 абзаца объясняют ценность и не тянут время;
- структура логична без повторов;
- каждый H2 раскрыт достаточно глубоко;
- тезисы видео не искажены;
- нет неподтверждённых обещаний, юридических рисков и запрещённых формулировок из `LEGAL.md`;
- ключи распределены естественно;
- текст звучит как человек, но без неряшливости;
- есть конкретика: шаги, примеры, ошибки, выводы;
- FAQ не повторяет основной текст дословно;
- SEO title/meta не кликбейтные и не длиннее лимитов.

Сохрани отчёт:

```text
RESULTS/<basename>-editor-review.md
```

Если правки были существенные, обнови `RESULTS/<basename>-article-final.md` после ревью.

## Шаг 9. Сгенерировать горизонтальную картинку через LidFly

Используй только MCP `lidfly`:

```js
mcp__lidfly__generate_ad_image({
  format: "lidfly_hero_1920x1080",
  prompt: "<полный промпт с композицией и короткими надписями>"
})
```

Не генерируй картинку сам и не накладывай текст локально. Надписи должны быть частью промпта к LidFly.

Промпт должен включать:

- формат: горизонтальная обложка 16:9, 1920×1080;
- стиль под тему статьи и бренд проекта;
- главный визуальный объект;
- фон, свет, палитру;
- 1-3 короткие надписи на картинке, например H1/главный оффер/лейбл урока;
- требование: крупная читабельная типографика, без мелкого текста;
- отсутствие лишних логотипов, водяных знаков и случайных букв.

Перед генерацией можно вызвать `get_recent_images({ limit: 10 })`, чтобы не плодить дубликаты. Если `generate_ad_image` вернул 504, не ретраить сразу: сначала проверить `get_recent_images`.

Сохрани URL и, если нужно, локальную копию:

```text
RESULTS/<basename>-cover-url.txt
RESULTS/<basename>-cover.png
```

## Шаг 10. Финальная выдача

В конце отдай кратко:

- путь к финальной статье;
- количество символов с пробелами и без;
- главный ключ и 5-10 поддерживающих;
- SEO title;
- meta description;
- путь к редакторскому ревью;
- URL обложки LidFly и локальный путь, если скачивал;
- что осталось проверить вручную, если есть риски.

Не вставляй всю статью в чат, если она длинная. Достаточно структуры, файлов и короткого summary.

## Критерии топ-качества

- Статья не пересказывает видео подряд, а превращает его в полезный веб-материал.
- В первом экране понятны тема, ценность и кому это нужно.
- Есть собственная редакторская логика: что главное, что второстепенно, какие ошибки читателя предупредить.
- Семантика влияет на структуру, а не просто добавлена списком в конец.
- Текст можно публиковать без стыда: нет AI-глянца, но и нет намеренной неряшливости.
- Обложка соответствует теме статьи и содержит короткий читаемый месседж.
