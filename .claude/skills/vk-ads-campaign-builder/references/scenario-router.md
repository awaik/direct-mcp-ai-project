# Scenario Router — что меняется под объект рекламы

Дефолт скилла — **сайт + конверсии** (пиксель, события, `site_conversions`). Этот файл — оверрайды: что в каждом сценарии меняется в цели, аналитике, обязательных проверках и рисках. Грузи на фазе 2, как только понял объект рекламы. Официальные VK-доки под сценарий — в `references/official-docs-map.md`, механика инструментов — в `references/mcp-vk-workflow.md`.

## Что остаётся во всех сценариях

Бриф (фаза 1), read-only + ID & facts ledger (фаза 3), план записи (фаза 4), preflight-gate (фаза 5) и пять правил `SKILL.md` — в силе всегда. Меняется только то, что ниже.

## Сайт / лиды / продажи (дефолт)

- Цель: `site_conversions`; `traffic` — только если событий мало и нужен прогрев данных.
- Обязательно: пиксель VK, целевые события, UTM, согласие на обработку ПД на посадочной.
- `priced_goal` на группе для `site_conversions`.
- 2-3 группы по разным аудиториям.

## Лид-форма

- ⚠ Через MCP формы сейчас в основном **читаются** (`vk_get_lead_forms`, `vk_get_leads`). Если создание/архивация формы не реализованы — не обещай live-create через MCP; подготовь ТЗ и используй существующую форму.
- Минимум полей, быстрый оффер, бонус/скидка.
- Обязательно: юр-данные, интеграция CRM/почта, регламент и скорость обработки заявок.
- Результат меряем через статистику лидов в MCP, не через CPC/CTR.

## Сообщество

- Цели: подписка / сообщение / продвижение поста.
- Проверь права на сообщество, объект, пост без рекламной пометки, креатив, юр-блок.
- Аудитория подписчиков: ссылки даёт пользователь (**не ищи сообщества через веб**); источник через `vk_get_vk_groups` / `vk_add_vk_group`, сегмент `remarketing_vk_group` (`params.source_id = object_id`, не внутренний `id`).

## Mini Apps / Senler

- `priced_event_type = 43`; `priced_goal.source_id = tracker.id`; `priced_goal.name = event.name`.
- Инструменты: `vk_get_mobile_apps`, `vk_get_inapp_events`, `vk_get_inapp_stats`.
- ⚠ Нужны событие `VKWebAppTrackEvent` и ясность, что считается результатом — иначе стоп (точка остановки).

## Ecommerce / каталог

- Каталог/фид + связь с пикселем или приложением.
- Обязательные события динамического ретаргетинга: просмотр карточки товара, `add_to_cart`, `purchase`; `product_id` совпадает с фидом, `value` передаёт стоимость.
- Тип: привлечение новых **или** динамический ретаргетинг — не смешивай в одной группе.

## Ретаргетинг / LAL / базы / контекстные фразы

- Отдельная гипотеза с отдельными сегментами и бюджетом, не «добавка» к холодной аудитории.
- Контекстные фразы: `vk_create_search_phrases` возвращает `segment_id` → передавай его в `targetings.segments`. Не используй несуществующие `search_phrases_id` / `context_phrases`.
- LAL и ретаргетинг строй на **качественных** событиях, а не на мусорных лидах.

## Охват / видео / брендинг

- Не смешивай с performance CPA. Метрики: CPM, охват, частота, video funnel, Brand/Search/Visit lift если доступно.

## Агентский режим

- Сначала определи клиента через `vk_get_agency_clients`, дальше последовательно передавай один и тот же `client_id` во все неагентские вызовы. Не смешивай свой кабинет и клиентский.
- Не создавай клиента и не делай `vk_transfer_funds` без отдельного подтверждения.

## VK-owned URL (для всех VK-объектов)

Лид-формы, сообщества, Mini Apps: используй `vk_resolve_url` / `vk_create_url`, **не** передавай строковый URL в низкоуровневые поля `urls`.
